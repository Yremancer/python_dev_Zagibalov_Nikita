# Тестовое задание — Python

## Запуск

Создать .env файл в корне проекта по примеру .env.example

```bash
docker compose up --build
```

API доступно по адресу: `http://localhost:8000`

Swagger документация: `http://localhost:8000/docs`

## API эндпоинты

### GET /api/comments?login={login}

Возвращает датасет комментариев пользователя: логин, заголовок поста, автор поста, количество комментариев.

```
GET /api/comments?login=bob
```

### GET /api/general?login={login}

Возвращает общую статистику действий пользователя по дням: количество входов, выходов, действий в блоге.

```
GET /api/general?login=alice
```

## Стек

- **Python 3.12**, **FastAPI** — асинхронный веб-фреймворк
- **SQLAlchemy** (async) + **aiomysql** — ORM и асинхронный драйвер MySQL
- **MySQL 8.4** — СУБД
- **Docker / Docker Compose** — контейнеризация

## Принятые решения

### Отхождения от задания

#### Добавлена таблица comment

В задании таблица `comment` не указана в схеме, но мне показалось, что она и ее данные были бы полезны в подобногго рода приложении. Таблица `comment` была добавлена в основную БД (`main`) с полями `id`, `text`, `author_id`, `post_id`. Это позволяет формировать датасет комментариев напрямую из основной БД без необходимости джойнить данные из двух баз данных.

#### Добавлен entity_id в таблицу logs

Поле `entity_id` добавлено в таблицу `logs` для хранения идентификатора сущности, с которой пользователь взаимодействовал (например, ID поста при создании или ID комментария). Без этого поля невозможно было бы связать запись в логе с конкретным объектом.

### Почему MySQL

В задании предлагалось использовать SQLite, MySQL или PostgreSQL. Был выбран MySQL, потому что он позволяет делать JOIN между таблицами разных баз данных (схем) в рамках одного сервера. Это решает проблему связей между объектами двух БД — можно выполнить один SQL-запрос вместо нескольких отдельных запросов с последующим маппингом данных на стороне приложения, что при большом объёме данных работало бы значительно медленнее.
